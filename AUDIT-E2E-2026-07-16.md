# Audit End-to-End SPDM — 16 Julai 2026

**Commit penutup:** `63d8e9c` (`fix: harden multi-tenant document workflows`)

## Keputusan eksekutif

Kod dan aliran tempatan SPDM telah diaudit melalui pemeriksaan route/policy/service, ujian Pest, dan Google Chrome sebenar melalui Playwright. Pengasingan tenant MAM/MAN, matriks sembilan peranan, intake UI/WhatsApp/e-mel, OCR, carian, minit, maklum balas, susulan, kelulusan, retensi, billing dan superadmin termasuk dalam skop.

Keputusan semasa:

- aliran tempatan dan regression suite **lulus** berdasarkan bukti penutup dalam bahagian “Bukti ujian”;
- sistem **belum boleh diisytihar live production** kerana host/domain SPDM, secret silang WhatsApp, IMAP/SMTP/COS/Redis/PostgreSQL/Meilisearch production dan restore drill belum mempunyai bukti;
- nama peranan rasmi ialah `nazir`, bukan `naqib`. Audit menggunakan Nazir sebagai peranan yang paling hampir. Jangan beri kuasa baharu kepada “Naqib” tanpa keputusan pemilik produk.

## Cara audit dijalankan

1. Inventori semua route aplikasi, resource, page, action, policy, model, service, job dan scheduler.
2. Semakan statik keselamatan PHP dan pemetaan tenant/sensitiviti.
3. Pangkalan data E2E berasingan di `storage/app/e2e-audit.sqlite` dengan tenant MAM dan MAN.
4. Google Chrome headless melalui Playwright untuk superadmin serta sembilan peranan tenant.
5. Setiap pautan sidebar yang benar-benar kelihatan bagi setiap peranan dibuka dan diperiksa.
6. Cubaan silang tenant MAM ke MAN dibuat bagi setiap peranan.
7. UAT servis kerani → pengerusi → setiausaha → admin → audit.
8. Dua imej sebenar pengguna dan satu PDF bertulis diproses melalui OCR, derived PDF dan carian.
9. Webhook WhatsApp dan e-mel plus-addressing diuji dengan kes positif, negatif, pendua dan silang tenant.

In-app Chrome MCP tidak mempunyai browser `iab` yang tersedia dalam sesi ini. Audit browser diteruskan dengan pemasangan Google Chrome sebenar pada mesin melalui Playwright; ini direkod secara jujur dan bukan dianggap sebagai MCP Chrome yang berjaya.

### Cara ulang ujian Chrome local

Gunakan terminal PowerShell khusus dan pastikan path DB di bawah tepat sebelum `migrate:fresh`; command itu hanya boleh ditujukan kepada DB E2E, bukan data sebenar:

```powershell
$env:DB_CONNECTION='sqlite'
$env:DB_DATABASE=(Join-Path (Get-Location) 'storage\app\e2e-audit.sqlite')
$env:QUEUE_CONNECTION='sync'
$env:CACHE_STORE='array'
$env:SCOUT_DRIVER='collection'
$env:MAIL_MAILER='log'
$env:IMAP_ENABLED='false'
$env:WHATSAPP_DRIVER='log'
$env:TESSDATA_PREFIX="$env:USERPROFILE\scoop\apps\tesseract-languages\current"
New-Item -ItemType File -Force $env:DB_DATABASE | Out-Null
php artisan migrate:fresh --seed
php artisan serve --host=127.0.0.1 --port=8092
```

Dalam terminal kedua:

```powershell
$env:SPDM_OCR_FIXTURE_1='C:\path\dokumen-1.jpeg'
$env:SPDM_OCR_FIXTURE_2='C:\path\dokumen-2.jpeg'
$env:SPDM_OCR_TERM_1='frasa unik dokumen 1'
$env:SPDM_OCR_TERM_2='frasa unik dokumen 2'
npm run test:e2e
```

Fixture sebenar berada di luar repositori dan tidak dicommit. Akaun demo/sembilan peranan datang daripada seeder; suite tidak boleh ditujukan kepada production.

## Workflow dokumen hujung-ke-hujung

### UI

1. Kerani atau peranan berizin buka Peti Masuk dan pilih `+ Muat Naik Dokumen`.
2. Sistem menyemak MIME, saiz 25 MB, kuota, keahlian tenant dan hash.
3. Nama fail asal disimpan; rekod `peti_masuk` dan media private dicipta.
4. `ProcessOcrJob(recordId, mosqueId)` memproses imej/PDF.
5. OCR menyimpan `ocr_text`, menghasilkan `searchable.pdf` dan mengindeks Peti Masuk.
6. Kerani semak tab OCR, pilih `Klasifikasikan`, isi metadata, pilih/cipta fail dan sensitiviti.
7. Sistem memperuntuk nombor kandungan secara atomik, mewarisi sensitiviti tertinggi dan mengira retensi.
8. Rekod tersedia untuk minit, approval, versi, pindah fail, QR, audit dan signed download.

Wizard `Rekod Baharu` memendekkan proses dengan terus muat naik dan klasifikasi. Isu nama fail rawak pada wizard/versi telah dibetulkan; nama asal kini dipelihara.

### WhatsApp

Setup admin tenant:

1. Tetapan Masjid → `Aktifkan WhatsApp`.
2. SPDM provision tenant gateway menggunakan external ID dan HMAC; API key disimpan terenkripsi.
3. `Pasangkan Nombor` → QR atau kod telefon → `Segerakkan Status` sehingga connected.
4. Ahli & Peranan → tetapkan nombor dan opt-in WhatsApp bagi setiap ahli dalam tenant itu.

Penggunaan:

1. Ahli hantar mesej tepat `spdm` ke nombor rasmi tenant.
2. Slot upload 10 minit dibuka untuk pasangan `session + phone` sahaja.
3. Ahli hantar satu dokumen; media dengan kapsyen keyword turut disokong.
4. Webhook HMAC menyemak session, integration connected, tenant aktif, ahli, kuota, MIME, message ID dan hash.
5. Dokumen hanya masuk Peti Masuk tenant session tersebut; tenant/nombor lain tidak boleh menggunakan slot itu.
6. Sistem membalas rujukan sementara dan memaklumkan pemegang inbox.

### E-mel multi-tenant

Satu mailbox platform yang menyokong plus-addressing digunakan, bukan satu mailbox bercampur tanpa identiti. Contoh username platform `scan.diwan@gmail.com` menghasilkan:

- MAM: `scan.diwan+mam@gmail.com`
- MAN: `scan.diwan+man@gmail.com`

Admin tenant mengaktifkan e-mel, memilih keyword (lalai `spdm`) dan allowlist pengirim. Scheduler setiap minit membaca `To`/`Delivered-To`, mengekstrak slug tepat, kemudian menyemak tenant aktif, toggle, allowlist, keyword dalam subjek/isi, MIME, kuota dan dedup. E-mel `+man` tidak boleh masuk MAM; allowlist MAM tidak terpakai kepada MAN.

Jika provider tidak menyokong plus-addressing, gunakan subdomain alias/catch-all yang tetap memetakan alias unik kepada slug; jangan route berdasarkan alamat `From` sahaja.

## Workflow minit, maklum balas dan kelulusan

1. Pengguna berizin buka rekod → `Edarkan Minit`.
2. Pilih penerima tindakan, s.k., arahan/catatan dan keutamaan biasa/segera/kritikal.
3. Sistem mengesahkan setiap penerima ialah ahli aktif tenant dan boleh melihat sensitiviti rekod.
4. SLA dijana 7/3/1 hari dan notifikasi dihantar melalui saluran aktif tenant.
5. Penerima melihat isi arahan terus dalam `Minit Saya`, bukan sekadar tajuk.
6. `Balas & Edarkan` mencipta minit anak (`parent_id`) supaya sejarah susulan kekal.
7. Setiap penerima tindakan pilih `Tanda Selesai`; hanya apabila semua selesai, minit menjadi selesai dan pengirim menerima notifikasi selesai.
8. `Mohon Kelulusan` hanya menawarkan pelulus aktif yang boleh melihat rekod.
9. Pelulus mesti mengesahkan kata laluan untuk Lulus/Tolak; masa, IP, nota dan sejarah keputusan disimpan.
